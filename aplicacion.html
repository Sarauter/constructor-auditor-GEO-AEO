Mentionate · Auditor GEO/AEO

Poner en marcha la auditoría por URL en Cloudflare Pages

Guía paso a paso para desplegar el proyecto y que la herramienta funcione con cualquier empresa.

Por qué hace falta este paso

La web ya está en GitHub, pero ahora mismo se sirve como una página estática. La función que audita las URLs (/api/audit) es serverless y solo se ejecuta en Cloudflare Pages. Por eso aparece el mensaje rojo «La función de auditoría no está disponible aquí».

Al conectar el repositorio de GitHub con Cloudflare Pages, esa función cobra vida y la auditoría empieza a funcionar. Como sarauter.com ya vive en esta misma cuenta de Cloudflare, todo queda en el mismo sitio.

1

Crear el proyecto en Cloudflare Pages

Conectar el repositorio de GitHub. Tarda un par de minutos.

  1. Entrar en dash.cloudflare.com (con la cuenta donde está sarauter.com).
  2. Menú lateral → Workers & Pages → botón Create application → pestaña Pages → Connect to Git.
  3. Autorizar GitHub y seleccionar el repositorio Sarauter/constructor-auditor-GEO-AEO.
  4. En la configuración de compilación (build):
    • Framework preset: None
    • Build command: dejar en blanco (vacío)
    • Build output directory: . (un punto) o dejarlo vacío — Cloudflare detecta wrangler.toml
  5. Pulsar Save and Deploy.
Al terminar dará una URL tipo constructor-auditor-geo-aeo.pages.dev. La auditoría de URLs ya funcionará ahí con los valores por defecto. El login y el envío de leads todavía no — para eso es el paso 2.
2

Configurar las variables y secretos

Para que funcionen el login de equipo y el envío de leads por email.

Dentro del proyecto: Settings → Variables and Secrets (o «Environment variables»). Añadir estas, marcando las de contraseña/clave como Secret / Encrypt:

VariableValorPara qué
OPTIMIZER_USER asaravia Usuario del login del optimizador
OPTIMIZER_PASSWORD secret (la contraseña que os pasó Andrea) Contraseña del login del optimizador
RESEND_API_KEY secret clave de resend.com Para que los formularios de contacto lleguen por email a marketing@sarauter.com
Sobre RESEND_API_KEY: hay que crear una cuenta gratuita en resend.com y verificar el dominio de envío (sarauter.com) para poder generar la clave. Si aún no la tenéis, se puede dejar para después: la auditoría funciona igual sin ella; lo único que no llegará por email son los leads del formulario.
Importante: si el proyecto usa entornos separados, añadir las variables tanto en Production como en Preview.
3

Volver a desplegar

Los secretos solo se aplican tras un nuevo despliegue.

  1. Ir a la pestaña Deployments del proyecto.
  2. En el último despliegue, menú ··· → Retry deployment.
  3. Alternativa: cualquier push nuevo a la rama main dispara un despliegue automático.
4

Probar que funciona

Comprobación final.

  1. Abrir https://tu-proyecto.pages.dev.
  2. Pegar cualquier URL de empresa (por ejemplo kstudio.com) y pulsar Auditar.
  3. Debe aparecer el diagnóstico con puntuaciones — ya no el mensaje rojo de error.
Si sale el diagnóstico, ¡está listo! La auditoría ya funciona con cualquier empresa.
5

Dominio propio (opcional)

Para verlo en un subdominio de sarauter.com en vez de la URL genérica .pages.dev.

  1. Dentro del proyecto de Pages → Custom domains → Set up a custom domain.
  2. Escribir el subdominio deseado, por ejemplo auditor.sarauter.com o app.sarauter.com.
  3. Como sarauter.com ya está en esta cuenta de Cloudflare, el registro DNS se crea automáticamente. Confirmar y esperar unos minutos.

Resumen rápido

  • Crear proyecto en Workers & Pages → Pages → Connect to Git → repo Sarauter/constructor-auditor-GEO-AEO
  • Build command vacío, output .
  • Añadir variables: OPTIMIZER_USER, OPTIMIZER_PASSWORD (secret), RESEND_API_KEY (secret, opcional al principio)
  • Retry deployment
  • Probar con una URL en *.pages.dev
  • (Opcional) Añadir subdominio de sarauter.com
